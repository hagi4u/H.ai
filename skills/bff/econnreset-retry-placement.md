# ECONNRESET 리트라이, 어디에 둘 것인가?

> 문제 해결 과정에서 "리트라이를 BFF에 둘지, axios 클라이언트에 둘지" 고민한 기록.
> 결론보다 **판단 근거**에 초점을 맞춤.

## 배경

- Pay BE가 blue/green 배포될 때 기존 keep-alive 소켓이 RST 패킷으로 끊김
- BFF의 `fetch()`가 `TypeError: ECONNRESET`을 던짐
- 리트라이 없이는 Next.js가 이를 500으로 브라우저에 응답

## 고민: axios interceptor(브라우저)에서 리트라이하면 안 되나?

브라우저 → BFF → upstream 구조에서, 브라우저 axios에 리트라이를 넣으면 BFF가 500을 줬을 때 다시 시도하니까 동작은 한다.

### 하지만 문제가 있다

**1. 에러 원인을 구분할 수 없다**

브라우저 입장에서 BFF가 준 500이:
- ECONNRESET 때문인지 (리트라이 O)
- 진짜 서버 에러인지 (리트라이 X)

구분이 불가능하다. 에러 코드를 응답에 실어 보내는 방법도 있지만, 그건 BFF에 로직을 추가하는 것이므로 본질적으로 같다.

**2. 네트워크 홉이 불필요하게 늘어난다**

| 리트라이 위치 | 재시도 경로 |
|-------------|-----------|
| BFF (현재) | BFF → upstream만 재시도 |
| 브라우저 axios | 브라우저 → BFF → upstream 전체 재시도 |

서버 간 통신은 ms 단위인데, 브라우저 왕복은 수십~수백ms가 추가된다.

**3. POST 등 비멱등 메서드의 위험**

모든 500에 리트라이하면 POST 요청도 재시도될 수 있다. BFF에서는 ECONNRESET인 경우만 골라서, GET처럼 멱등한 메서드만 리트라이할 수 있다.

## 판단 원칙: 원인을 아는 곳에서 처리한다

> **에러의 원인을 식별할 수 있는 계층**이 리트라이 책임을 가져야 한다.

- BFF는 `TypeError` 메시지에서 `ECONNRESET`을 직접 확인할 수 있음
- 브라우저는 HTTP 500만 볼 수 있고, 그 원인을 알 수 없음

따라서 ECONNRESET 리트라이는 **BFF에 두는 것이 맞는 위치**.

## 결론

- BFF route에 `fetchWithRetry` 구현 (GET 메서드, ECONNRESET/ETIMEDOUT만 대상)
- 향후 멱등성이 보장되는 다른 메서드는 `RETRYABLE_METHODS`에 추가 가능
- axios interceptor 리트라이는 이 문제의 해결책으로는 부적합 (다른 용도로는 별개)

## 일반화할 수 있는 교훈

1. **리트라이 위치는 에러 식별 능력으로 결정한다** — 원인을 모르면 리트라이 조건도 정할 수 없다
2. **"동작은 한다"와 "올바른 위치다"는 다르다** — 브라우저 리트라이도 동작하지만, 부작용과 비효율이 따른다
3. **레이어 간 정보 비대칭을 인식한다** — 상위 레이어일수록 에러의 구체적 원인을 알기 어렵다
